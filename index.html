线程池优雅关闭
2025/6/23

这个对线程池的一个优化，主要是防止在宕机过程中丢失数据的一个问题，然后宕机其实也分为就是

  • 普通宕机（服务崩溃、系统重启、kill -15）这种
  • 强制宕机（如kill -9 、停电）这种

面对不同宕机我们要有不同的处理方式，因为首先在普通宕机的情况下，我们可以触发ShutDownHook函数，所以我们可以在CAP中的CA还衡量并选择。CAP指的是什么？

  • C指的是在宕机过程中我们依旧可以让数据不错乱，保持逻辑的完整，简单来说就是在宕机时尽量让程序跑完，如果要超时了我们持久化未完成的任务
  • A指的是在宕机过程中我们快速把未完成的任务存储到redis，核心是 最小化恢复时间，通过保存轻量级元数据（如任务ID、进度指针）实现。
  • P指的是物理存储方面的可靠性，比如存储到磁盘

强制宕机的情况下，我们能做的就是每一步写好日志BufferedWriter ，恢复之后我们自已去排查，强制宕机情况下,P是我们必须要保证的，如果连磁盘都不能持久化，CA何从谈起呢。

我们最终选择了CP，然后我们主要说一下这个普通宕机的这种情况，首先的话

import java.util.concurrent.ExecutorService;

public class GracefulShutdownManager {

private final ExecutorService executor;

private final TaskStatePersistence persistence;

public GracefulShutdownManager(ExecutorService executor, TaskStatePersistence persistence) {

​ this.executor = executor;

​ this.persistence = persistence;

}

public void registerShutdownHook() {

​ Runtime.getRuntime().addShutdownHook(new Thread(() -> {

​ System.out.println(“[ShutdownHook] 收到停机信号，开始优雅关闭…”);

​ initiateShutdown();

​ }));

}

private void initiateShutdown() {

​ // 1. 停止接收新任务

​ executor.shutdown();

​

​ try {

​ // 2. 等待现有任务完成

​ if (!executor.awaitTermination(30, TimeUnit.SECONDS)) {

​ // 3. 超时后强制关闭并持久化未完成任务

​ List pendingTasks = executor.shutdownNow();

​ persistence.savePendingTasks(pendingTasks);

​ }

​ } catch (InterruptedException e) {

​ Thread.currentThread().interrupt();

​ }

}

}

执行一条sql请求的全过程

1.首先要建立连接，如果是本地连接就用文件协议，如果是远程则是TCP/IP协议，通过端口号，ip地址，账号密码连接

2.查询缓存，如果缓存命中则直接返回。Mysql8.0取消了这个模块

3.解析器：经过词法分析和语法分析构建语法树，语法树的结构就是 select -> c1,c2 -> from ->..

4.sql执行过程：首先是预处理阶段，检查表和字段是否存在，然后做一些处理，比如select * 的* 展开为全部的列

其次是优化阶段，将sql语句进行优化 比如where id > 10+6 优化为 where id > 16

最后一步是查询结果集调用存储引擎

什么是索引？

索引相当于书籍的目录，加快查询效率。

没有使用索引的情况下我们需要进行全盘扫描，时间复杂度是O（n），如果使用了索引我们在查询的时候可以使用二分查找，时间复杂度就是O（logn），因为mysql默认索引是B+树索引，他的搜索复杂度是O(logdN)d代表着树的高度。

索引分为哪几个

  • 数据结构：B+ tree、Hash索引、Full-text索引
  • 物理存储：聚簇索引（主键索引）、 二级索引
  • 字段特性：主键索引、唯一索引、普通索引、前缀索引
  • 字段个数：单列索引、多列索引

这个情况主要是真针对黑名单的，就是说我们最开始使用了本地缓存，然后从数据库里面不断刷DB，一分钟刷一次，最开始我们可以接收半分钟左右的延迟，不是那么高的一致性，随着业务发展，我们发现半分钟的延迟我们已经接受不了了，所以我们从本地缓存移动到了redis中，然后我们的redis数据结构采取的是String，key是用户id，value就是1.

然后就是本地缓存他是靠我们定时的去刷取DB的，这种情况必然会导致有一定时间的数据不一致性问题，当我换成redis时候，就可以完全避免这种问题了。

就是当我们查询的时候先查redis，如果有就返回，没有的话我们就再去查一遍数据库嘛，然后在redis中刷回来，这是一个读多写少的任务。但如果在修改的时候我们采取先更新数据库在删除缓存。

什么是RAG?

RAG全称是检索增强生成

  • 检索的意思是将用户提问的话进行编码，编码成向量，然后去向量数据库里面进行查询和检索
  • 增强是将检索到的文档数据进行一个拼接
  • 生成是大模型和上下文进行结合最后回答的话

问题："量子纠缠的实际应用有哪些？" 检索到文档：量子通信、量子计算相关论文片段 生成答案："量子纠缠应用于量子密钥分发（QKD）和量子计算机中的量子比特耦合..."

为什么会有RAG？

因为大模型会有知识局限性、幻觉问题、数据安全性问题

  • 知识局限性：大模型里面数据基本上都是现在网络中能查到的，有些非公开的和离线的数据差不多。

  • 幻觉：因为大模型的底层基本上都是一些数学的运算，所以大模型很有可能会出现胡说八道的现象，尤其是在面对大模型的知识盲区

  • 数据安全性：很多企业内部的数据是不会对外公开的，所以大模型是搜不到的

    可以将RAG的核心理解为“检索+生成”，前者主要是利用向量数据库的高效存储和检索能力，召回目标知识；后者则是利用大模型和Prompt工程，将召回的知识合理利用，生成目标答案。

    完整的RAG应用流程主要包含两个阶段：

    • 数据准备阶段：数据提取——>文本分割——>向量化（embedding）——>数据入库
    • 应用阶段：用户提问——>数据检索（召回）——>注入Prompt——>LLM生成答案

什么是？

大家都知道事务有四大特性（ACID）：

  • 原子性（Atomicity）
  • 一致性（Consistency）
  • 隔离性（Isolation）
  • 持久性（Durability）

分布式事务是指跨多个数据库或服务的操作，需要保证所有参与方要么全部成功提交，要么全部回滚。

核心挑战：在分布式环境下，网络分区、节点故障等问题会破坏ACID特性。

分布式事务的定理

CAP

由Eric Brewer提出，描述分布式系统的三个核心特性：

特性 描述
Consistency 所有节点同时看到相同数据
Availability 每次请求必须得到响应
Partition Tolerance 网络分区时系统仍能运行

CAP不可兼得：实际系统中必须牺牲其一（通常选择CP或AP）。

CAP取舍问题

由于分区容错性（P）是分布式系统的刚需，实际取舍在C和A之间：

选择 场景 案例
CP 强一致性优先 银行转账、ZooKeeper
AP 高可用性优先 电商库存、Cassandra

Base定理

对CAP的补充，强调最终一致性：

  • BA（Basically Available）：允许降级服务
  • S（Soft State）：允许中间状态不一致
  • E（Eventual Consistency）：最终达成一致

适用于容忍延迟一致性的场景（如社交媒体的点赞数同步）。

刚性事务

2PC（两阶段提交）

阶段1：准备阶段

  1. 协调者发送Prepare请求
  2. 参与者执行事务但不提交
  3. 返回Ready/Abort状态

阶段2：提交阶段

  • 全部Ready → 发送Commit
  • 任意Abort → 发送Rollback
Coordinator → Participant 1: Prepare Request
Participant 1 → Coordinator: Prepare OK
Coordinator → Participant 2: Prepare Request
Participant 2 → Coordinator: Prepare OK
Coordinator → All: Commit Request

什么是Raft？

​ Raft是一种非拜占庭的算法，指的是所有通信都必须是真实的而非捏造的，Raft最多容许N个节点中（N为奇数）故障(N-1)/2,就是不能故障超过半数。

​ 如图所示，这是Raft的日志复制流程，每个节点都通过日志列表来记录对应的操作，Raft里面有三个角色，分别是Leader（领导）、Follower（跟随者）、Candidate（获选人）。最开始每一个都是Follower，通过Raft的定时器来变成Candidate，之后通过投票选举变成Leader。每一个客户端都要和Leader交互。

​ 当Leader接收到请求之后，并不会立刻执行对应操作，而是将操作写入到日志文件当中，并发送给Follower让他们也写入到自已的日志中并返回信息给Leader，只有超过半数写入日志后Leader才会返回给客户端，同时告诉其他节点进行此操作。

Raft解决了什么问题？

​ Raft是一致性算法，所谓一致性算法就是解决再单机情况下如果出现问题，导致数据丢失等事情的发生。那如何解决的呢？答案很简单，就是通过备份，在一台机器上容易发生问题？那我就多台，在每台机器上都执行对应的操作不就行了！

Raft的选举机制

编辑

​ 如图所示，这就是Raft的初次选举，在最开始的时候所有节点都是Follower，每个Follower都有一个定时器，当收到Leader的信息之后会更新定时器，当定时器结束时没有收到Leader发来消息，证明Leader已经死亡或者根本不存在，这个时候该节点就会变成Candidate，通过投票选举（该节点向其他节点发出投票信息，如果超过一半以上即通过选举，该节点成为Leader），成功成为Leader之后，新的Leader会向其他Follower发送心跳信息，方式其他节点成为Leader节点。

​ 如果该节点未能成功竞选为Leader，那么Raft会重复这个操作，直到出现Leader为止。

选举分区

​ 选举分区是一种特殊情况，指的是在Follower节点和Leader发生沟通问题，从而自已形成了一个区域，并且在这个区域内会产生一个新的Leader。但是因为规定，必须要写入日志的节点要超过一半才可以，所以N为奇数时候，无论怎么分区都是可以正常运行的。

Raft的日志复制

日志追加不会覆盖
