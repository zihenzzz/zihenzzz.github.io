DDD

OK，我们先来说一下什么是DDD，DDD是领域驱动设计，每个公司的使用方式肯定是不一样的，然后我来说一下我的个人理解。首先说一下贫血模型和充血模型，正常mvc架构下的都是贫血模型，就是对象都是属性没有具体的方法和动作这种。

举个例子，比如我们从数据库中查询数据然后扔到redis中，让它下一次查询更快，如果我们用贫血模型的话，我们就要把key的处理位置放在服务层，就是一个公共的service层里面嘛，然后我们需要拼两次这个key，一次是放一次是查。如果我们用充血模型，在对象里面直接加入拼接key的方法，让他在对象里面就可以拼接。对吧，如果你每个的service需要这个key，那么你每一个service都要去写一个拼装的逻辑，这种把领域对象之内的事情分给各个service就叫做贫血。

传统mvc层：controller service mapper

DDD：api/表现层 application domain infra

DDD在修改或者添加业务的时候 修改领域对象就可以，但是MVC需要修改service

AQS
2025/6/26

AQS是并发工具类的框架，并发工具类是用来解决并发情况下，多个线程安全的去访问资源，那线程在访问的时候就要去判断，当前资源是否被占用，如果被占用了，那我们就只能阻塞、重试、放弃这三种选项，如果没有被占用，那就可以访问。

那我们如何表示共享资源是否被占用呢？我们需要一个state变量来表示，0代表没有被访问，1代表正在被访问、占用。这也是ReentrantLock的思想，那state设置成布尔类型会不会好一点呢？答案是不可以的，

因为有的资源最多支持三个线程访问，这个就是Semaphore 的思想

还有情况比如当前线程需要多次访问资源，访问一次state+1，释放一次-1，这个就是ReentrantLock可重入的思想。

当然，多个线程修改state要保证可见性和原子性，可见性由volatile修饰，原子性是通过cas的unsafe的compareandswapInt是保证。

如果访问失败了呢？一直自旋吗？当然不是，因为频繁的自旋会浪费cpu的，所以我们的做法应该是应该让访问失败的线程阻塞住，等资源空闲的再去唤醒.

但是被阻塞的资源有很多啊，所以我们需要创建一个先进先出的阻塞队列，把线程包装成一个一个Node节点，用节点去组成一个双向链表。这个就是CLH阻塞队列的思想。

线程池优雅关闭
2025/6/23

这个对线程池的一个优化，主要是防止在宕机过程中丢失数据的一个问题，然后宕机其实也分为就是

  • 普通宕机（服务崩溃、系统重启、kill -15）这种
  • 强制宕机（如kill -9 、停电）这种

面对不同宕机我们要有不同的处理方式，因为首先在普通宕机的情况下，我们可以触发ShutDownHook函数，所以我们可以在CAP中的CA还衡量并选择。CAP指的是什么？

  • C指的是在宕机过程中我们依旧可以让数据不错乱，保持逻辑的完整，简单来说就是在宕机时尽量让程序跑完，如果要超时了我们持久化未完成的任务
  • A指的是在宕机过程中我们快速把未完成的任务存储到redis，核心是 最小化恢复时间，通过保存轻量级元数据（如任务ID、进度指针）实现。
  • P指的是物理存储方面的可靠性，比如存储到磁盘

强制宕机的情况下，我们能做的就是每一步写好日志BufferedWriter ，恢复之后我们自已去排查，强制宕机情况下,P是我们必须要保证的，如果连磁盘都不能持久化，CA何从谈起呢。

我们最终选择了CP，然后我们主要说一下这个普通宕机的这种情况，首先的话

import java.util.List;

import java.util.concurrent.ExecutorService;

public class GracefulShutdownManager {

private final ExecutorService executor;

private final TaskStatePersistence persistence;

public GracefulShutdownManager(ExecutorService executor, TaskStatePersistence persistence) {

​ this.executor = executor;

​ this.persistence = persistence;

}

public void registerShutdownHook() {

​ Runtime.getRuntime().addShutdownHook(new Thread(() -> {

​ System.out.println(“[ShutdownHook] 收到停机信号，开始优雅关闭…”);

​ initiateShutdown();

​ }));

}

private void initiateShutdown() {

​ // 1. 停止接收新任务

​ executor.shutdown();

​

​ try {

​ // 2. 等待现有任务完成

​ if (!executor.awaitTermination(30, TimeUnit.SECONDS)) {

​ // 3. 超时后强制关闭并持久化未完成任务

​ List pendingTasks = executor.shutdownNow();

​ persistence.savePendingTasks(pendingTasks);

​ }

​ } catch (InterruptedException e) {

​ Thread.currentThread().interrupt();

​ }

}

}

执行一条sql请求的全过程

1.首先要建立连接，如果是本地连接就用文件协议，如果是远程则是TCP/IP协议，通过端口号，ip地址，账号密码连接

3.解析器：经过词法分析和语法分析构建语法树，语法树的结构就是 select -> c1,c2 -> from ->..

4.sql执行过程：首先是预处理阶段，检查表和字段是否存在，然后做一些处理，比如select * 的* 展开为全部的列

其次是优化阶段，将sql语句进行优化 比如where id > 10+6 优化为 where id > 16

最后一步是查询结果集调用存储引擎

什么是索引？

索引相当于书籍的目录，加快查询效率。

Please let me know if there is any other problem.

没有使用索引的情况下我们需要进行全盘扫描，时间复杂度是O（n），如果使用了索引我们在查询的时候可以使用二分查找，时间复杂度就是O（logn），因为mysql默认索引是B+树索引，他的搜索复杂度是O(logdN)d代表着树的高度。

索引分为哪几个

  • 数据结构：B+ tree、Hash索引、Full-text索引
  • 物理存储：聚簇索引（主键索引）、 二级索引
  • 字段特性：主键索引、唯一索引、普通索引、前缀索引
  • 字段个数：单列索引、多列索引

什么是RAG?

RAG全称是检索增强生成

  • 检索的意思是将用户提问的话进行编码，编码成向量，然后去向量数据库里面进行查询和检索
  • 增强是将检索到的文档数据进行一个拼接
  • 生成是大模型和上下文进行结合最后回答的话

问题："量子纠缠的实际应用有哪些？" 检索到文档：量子通信、量子计算相关论文片段 生成答案："量子纠缠应用于量子密钥分发（QKD）和量子计算机中的量子比特耦合..."

为什么会有RAG？

因为大模型会有知识局限性、幻觉问题、数据安全性问题

  • 知识局限性：大模型里面数据基本上都是现在网络中能查到的，有些非公开的和离线的数据差不多。

  • 幻觉：因为大模型的底层基本上都是一些数学的运算，所以大模型很有可能会出现胡说八道的现象，尤其是在面对大模型的知识盲区

  • 数据安全性：很多企业内部的数据是不会对外公开的，所以大模型是搜不到的

    可以将RAG的核心理解为“检索+生成”，前者主要是利用向量数据库的高效存储和检索能力，召回目标知识；后者则是利用大模型和Prompt工程，将召回的知识合理利用，生成目标答案。

    完整的RAG应用流程主要包含两个阶段：

    • 数据准备阶段：数据提取——>文本分割——>向量化（embedding）——>数据入库
    • 应用阶段：用户提问——>数据检索（召回）——>注入Prompt——>LLM生成答案
